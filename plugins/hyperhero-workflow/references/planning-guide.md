# 規劃階段詳細指南

> **強制要求**: 所有開發任務必須使用 devteam skill 進行規劃

---

## 目標

使用 devteam skill 進行團隊協作規劃，建立清晰的任務分配和執行計畫，確保開發流程高效、有序。

---

## devteam Skill 使用方式

### 1. 啟動規劃會議

**命令**:
```typescript
task(
  category="unspecified-high",
  load_skills=["devteam"],
  prompt="規劃 [具體任務描述]。

  【規劃範圍】
  1. 任務拆解：將任務分解為可執行的子任務
  2. 技術決策:確認使用的技術棧和架構
  3. 角色分配:前端、後端、測試等角色分配
  4. 時程規劃:各階段預估時間
  5. 魚險識別:潛在風險和應對措施

  【輸出要求】
  - 詳細的任務清單
  - 技術架構圖
  - 開發時程表
  - 風險評估報告"
)
```

### 2. 規劃內容詳解

#### 2.1 任務拆解

**目標**: 將大任務分解為可獨立執行的子任務（1-3 小時可個）

**拆解原則**:
- **單一職責**: 每個子任務只負責一個明確的職責
- **可測試**: 每個子任務都有明確的驗收標準
- **獨立性**: 子任務之間盡量減少依賴

**拆解步驟**:
1. **識別主要功能模組**
   - 核心功能 (必須)
   - 次要功能 (加分項)
   - 鄰接功能 (可選)

2. **拆分技術層**
   - 前端層 (UI 組件)
   - 後端層 (API 端點)
   - 資料層 (資料庫)
   - 基礎設施 (配置、部署)

3. **分配任務**
   - 每個子任務分配給適當的技術層
   - 設定依賴關係
   - 識別關鍵路徑

**範例**:
```
任務：實作使用者認證系統

拆解：
├── 前端
│   ├── 登入頁面組件
│   ├── 註冊頁面組件
│   └── 表單驗證工具
├── 後端
│   ├── POST /auth/register
│   ├── POST /auth/login
│   ├── JWT 驗證中間件
│   └── 密碼加密服務
└── 資料層
    ├── User 表結構設計
    └── Migration 腳本
```

#### 2.2 技術棧決策

**目標**: 籭據專案需求選擇最適合的技術棧

**決策要點**:
- **團隊熟悉度**: 團隊是否熟悉該技術？
- **生態系成熟度**: 該技術是否穩定、有長期支援?
- **效能需求**: 該技術是否滿足效能要求?
- **擴展性**: 該技術是否易於擴展?

**常見選擇**:
| 選擇 | 前端框架 | 後端框架 | 資料庫 |
|------|----------|----------|--------|
| **高性能** | React + Next.js | Node.js + Express | PostgreSQL |
| **快速開發** | Vue + Nuxt | Python + FastAPI | MongoDB |
| **企業級** | Angular | Java + Spring Boot | MySQL |
| **輕量級** | Svelte | Go + Gin | SQLite |

#### 2.3 角色分配
**目標**: 為每個子任務分配合適的角色

**角色類型**:
- **前端工程師**: UI/UX 實作、組件開發
- **後端工程師**: API 開點開發、資料庫設計
- **測試工程師**: 單元測試、E2E 測試
- **DevOps 工程師**: 容器化、部署配置
- **技術負責人**: 架構設計、Code Review

**分配原則**:
- 根據技能分配
- 考慮工作量平衡
- 考慮時區差異
- 設置備用資源

#### 2.4 時程規劃
**目標**: 建立現實可行的開發時程

**規劃方法**:
- **樂觀估計**: 一切順利的情況 (風險較高)
- **正常估計**: 包含緩衝時間 (推薦)
- **悲觀估計**: 考慮所有風險 (最保守)

**時間分配**:
| 階段 | 佔比 | 說明 |
|------|------|------|
| 規劃 | 10% | 需求分析、架構設計 |
| 開發 | 50% | 編碼、測試、除錯 |
| 測試 | 20% | 單元測試、整合測試 |
| 部署 | 10% | 容器化、部署配置 |
| 優化 | 10% | 效能調整、重構 |

**緩衝時間**: 節預留 20-30% 的緩衝時間應對:
- 需求變更
- 技術難題
- Bug 修復

- Code Review 延遲

#### 2.5 魯險識別
**目標**: 識別潛在風險並制定應對措施
**風險類型**:
- **技術風險**: 使用新技術、技術難題
- **時間風險**: 時程緊迫、資源不足
- **需求風險**: 需求不明確、頻繁變更
- **整合風險**: 第三方依賴、系統整合
- **安全風險**: 敏感資料、權限控制

**應對措施**:
- **技術風險**: 技術預研、原型驗證
- **時間風險**: 優先級排序、功能裁剪
- **需求風險**: 需求確認、迭代開發
- **整合風險**: 早期整合、模擬測試
- **安全風險**: 安全審查、滲透測試

## 規劃輸出模板

### 1. 任務清單
```markdown
## 任務清單

### 核心任務
- [ ] [任務 1]: [描述] - [負責人] - [預估時間]
- [ ] [任務 2]: [描述] - [負責人] - [預估時間]
- [ ] [任務 3]: [描述] - [負責人] - [預估時間]

### 次要任務
- [ ] [任務 4]: [描述] - [負責人] - [預估時間]
- [ ] [任務 5]: [描述] - [負責人] - [預估時間]

### 可選任務
- [ ] [任務 6]: [描述] - [負責人] - [預估時間]
```

### 2. 技術架構
```markdown
## 技術架構
### 前端
- **框架**: [React/Vue/Angular/Svelte]
- **UI 庫**: [Material-UI/Ant Design/Element Plus]
- **狀態管理**: [Redux/Vuex/Zustand/Pinia]
- **樣式方案**: [CSS Modules/Tailwind/Styled Components]

### 後端
- **框架**: [Express/FastAPI/Gin/Spring Boot]
- **資料庫**: [PostgreSQL/MySQL/MongoDB]
- **ORM**: [Prisma/TypeORM/Sequelize]
- **認證**: [JWT/Session/OAuth]
### 基礎設施
- **容器**: [Docker/Podman]
- **反向代理**: [Nginx/Traefik]
- **監控**: [Prometheus/Grafana]
- **日誌**: [ELK/Loki]
```
### 3. 時程規劃
```markdown
## 驗收標準
### 功能驗收
- [ ] 所有核心功能已實作
- [ ] 所有 API 端點正確
- [ ] 所有 UI 頁面完整
### 品質驗收
- [ ] 測試覆蓋率 ≥ 80%
- [ ] 無安全漏洞
- [ ] 效能符合標準
- [ ] 無障礙標準符合
### 文件驗收
- [ ] API 文件完整
- [ ] 部署文件完整
- [ ] 使用者文件完整
```

## 規劃最佳實踝
### 1. 充分利用 devteam skill
- 讓團隊參與規劃過程
- 使用 devteam 的協作功能進行任務分配
- 定期使用 devteam 進度追蹤
### 2. 保持彈性
- 規劃不是一成不變的
- 定期檢視和調整規劃
- 保持通訊暢通
### 3. 文檔化
- 記錄所有決策過程
- 保存規劃文檔
- 定期更新規劃
### 4. 風險管理
- 主動識別風險
- 制定應對計畫
- 定期檢視風險狀態
